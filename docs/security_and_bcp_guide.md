## 2. 5つの安心ポイント

### ポイント1: データは日本国内（東京）に保存

| 質問 | 回答 |
|------|------|
| データはどこにあるの？ | 東京のGoogleデータセンターです |
| 海外に出ることはある？ | ありません。日本国内のみです |
| なぜ重要？ | 日本の法律で守られます |

**お客様への説明**:
```
「御社のデータは東京のデータセンターに保存されます。
 海外のサーバーに出ることはありませんので、
 日本の個人情報保護法の範囲内で管理されます。」
```

---

### ポイント2: 暗号化は「二重」

| どこで？ | 暗号化方式 | 素人向け説明 |
|---------|-----------|-------------|
| 保存時 | AES-256 | 金庫に入れて鍵をかけている状態 |
| 通信時 | TLS 1.3 | 届けるときも鍵付きの箱で運ぶ |

**お客様への説明**:
```
「データは保存するときも、やり取りするときも、
 すべて暗号化されています。
 万が一データが盗まれても、解読は事実上不可能です。
 銀行のオンラインバンキングと同じ仕組みです。」
```

---

### ポイント3: 会社ごとに完全分離（他社のデータは絶対見えない）

| 質問 | 回答 |
|------|------|
| 他の会社のデータが見えることはある？ | 絶対にありません |
| どうやって分けてる？ | アプリケーション側で必ず会社IDをチェックし、データベース側でも段階的に保護を強化中です |
| 技術名は？ | アプリレベルのアクセス制御（実施中）+ Row Level Security（RLS、段階的に適用中） |

**お客様への説明**:
```
「A社のデータをB社が見ることは、システム上不可能です。
 アプリケーション側で必ず会社IDをチェックする仕組みがあり、
 さらにデータベース側でも段階的に保護を強化しています。
 マンションの部屋が完全に分かれているのと同じで、
 隣の部屋には絶対に入れない設計になっています。」
```

> **技術補足（v10.55追加）**: Row Level Security（RLS）はPhase 4Aで全テーブルに完全実装予定。
> 現時点ではアプリレベルでの厳格なアクセス制御を実施しており、RLSは段階的に適用中。
> Phase 3で主要テーブル、Phase 4Aで全テーブルへ展開する計画です。

## 3. 厄介なお客さんQ&A（想定質問と回答）

### Q1: 「Googleって海外企業でしょ？大丈夫なの？」

**回答**:
```
「はい、Googleは米国企業ですが、データは日本国内（東京）に
 保存されます。また、Google Cloud は日本政府の『ISMAP』
 という認定を取得しており、政府機関も使用しています。

 日本のメガバンクや大手企業も Google Cloud を使っています。」
```

**根拠**:
- ISMAP登録済み（政府情報システムのためのセキュリティ評価制度）
- 三菱UFJ銀行、みずほ銀行なども Google Cloud を採用

---

### Q2: 「地震が来たらデータ消えない？」

**回答**:
```
「東京のデータセンターが被災しても、大阪にもコピーがあります。
 Googleのデータセンターは、震度7の地震にも耐える設計です。
 また、データは複数の場所に自動でコピーされているので、
 1箇所が壊れても、他の場所から復旧できます。」
```

**技術的補足**:
- リージョン: asia-northeast1（東京）
- 自動レプリケーション: 同一リージョン内で3箇所に複製
- オプション: 大阪（asia-northeast2）へのクロスリージョンレプリカ可能

---

### Q3: 「停電したらどうなるの？」

**回答**:
```
「Googleのデータセンターには、非常用電源（UPS）と
 自家発電機があります。停電しても、サービスは止まりません。

 過去に東京で大規模停電がありましたが、
 Google Cloud は一度も止まっていません。」
```

---

### Q4: 「ハッキングされたらどうするの？」

**回答**:
```
「Googleには世界最高レベルのセキュリティチームがいます。
 24時間365日、世界中の攻撃を監視しています。

 また、私たちのシステムでも、
 ・不正アクセスの自動検知
 ・異常なログイン試行のブロック
 ・すべての操作の記録
 を行っています。

 万が一侵入されても、データは暗号化されているので、
 中身を見ることはできません。」
```

---

### Q5: 「御社が倒産したらデータはどうなるの？」

**回答**:
```
「万が一の場合でも、お客様のデータはお客様のものです。
 契約終了時には、データをエクスポート（書き出し）して
 お渡しすることができます。

 また、データは標準的な形式（CSV、JSON）で出力できるので、
 他のシステムへの移行も可能です。」
```

---

### Q6: 「社員が悪意を持ってデータを持ち出したら？」

**回答**:
```
「すべての操作は記録されているので、
 誰が何をしたかは完全に追跡できます。

 また、アクセス権限は最小限に設定しており、
 必要な人だけが必要なデータにアクセスできる設計です。

 さらに、大量データのダウンロードなど、
 異常な操作は自動で検知してアラートを出します。」
```

---

### Q7: 「セキュリティの認証とか取ってるの？」

**回答**:
```
「Google Cloud は以下の認証を取得しています：

 ・SOC 2 Type II（第三者監査で毎年セキュリティをチェック）
 ・ISO 27001（情報セキュリティの国際規格）
 ・ISO 27017（クラウドセキュリティの国際規格）
 ・ISMAP（日本政府が認めたクラウド）

 これらは銀行や政府機関と同じレベルの認証です。」
```

---

### Q8: 「具体的な復旧時間は？」

**回答**:
```
「障害の種類によりますが、目安は以下の通りです：

 ・軽微な障害（一部機能停止）: 数分〜1時間
 ・中程度の障害（サーバー故障）: 1〜4時間
 ・大規模災害（データセンター全壊）: 4〜24時間

 ただし、Google Cloud の過去の実績では、
 大規模障害でも99.95%の稼働率を維持しています。
 これは年間で約4時間しか止まらない計算です。」
```

## 5. 正直なリスクと対応策

### リスク1: Googleの大規模障害

| 項目 | 内容 |
|------|------|
| **リスク** | Google Cloud 全体が止まる可能性（極めて低い） |
| **過去実績** | 過去5年で大規模障害は2回（各数時間） |
| **影響** | サービス一時停止（データ消失はなし） |
| **対応策** | 大阪リージョンへの自動フェイルオーバー（オプション） |
| **コスト** | +¥8,000/月 |

### リスク2: サイバー攻撃

| 項目 | 内容 |
|------|------|
| **リスク** | ハッキングによるデータ漏洩 |
| **対応策（実施済み）** | 暗号化、アクセス制御、監査ログ |
| **対応策（追加可能）** | WAF（Webアプリケーションファイアウォール） |
| **コスト** | +¥3,000/月 |

### リスク3: 人的ミス（誤操作）

| 項目 | 内容 |
|------|------|
| **リスク** | 社員が誤ってデータを削除 |
| **対応策（実施済み）** | 毎日のバックアップから復旧可能 |
| **対応策（追加可能）** | 削除前の確認ステップ追加 |
| **コスト** | 追加コストなし |

### リスク4: 長期的なコスト増

| 項目 | 内容 |
|------|------|
| **リスク** | データ量増加に伴うコスト増 |
| **現在** | 10GB → 月額¥17,000 |
| **5年後予想** | 50GB → 月額¥35,000 |
| **10年後予想** | 100GB → 月額¥50,000 |
| **対応策** | 古いデータの自動アーカイブで抑制可能 |
